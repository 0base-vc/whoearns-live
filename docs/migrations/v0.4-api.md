# API Migration - v0.4 Live Trend Leaderboard

v0.4 changes the default leaderboard from a latest-final-epoch view to a live
trend view and adds database-only validator search.

## Leaderboard Defaults

Before v0.4:

```text
GET /v1/leaderboard
```

ranked the most recent closed epoch with the legacy performance metric.

After v0.4:

```text
GET /v1/leaderboard
```

is equivalent to:

```text
GET /v1/leaderboard?window=live_trend&sort=income_per_slot
```

`live_trend` combines the current epoch's elapsed leader slots with the latest
closed epoch. Running-epoch values remain lower bounds until the epoch closes.

## Windows

- `live_trend` - current epoch elapsed leader slots + latest final epoch.
- `current_only` - current epoch elapsed leader slots only.
- `stable_trend` - current epoch elapsed leader slots + two latest final epochs.
- `final_epoch` - latest final epoch, or an explicit closed epoch with `epoch=N`.
- `decade_epoch` - latest complete 10-epoch block. Validators must have all
  10 epoch rows to rank.

## Compatibility

Existing `/v1` clients keep working for common legacy forms:

- `GET /v1/leaderboard?epoch=N` is treated as
  `GET /v1/leaderboard?window=final_epoch&epoch=N`.
- `sort=performance` and `sort=income_per_stake` map to
  `sort=income_per_slot`.
- `sort=median_fee` maps to `sort=fees`.

Clients that need the old "latest closed epoch only" behavior should request:

```text
GET /v1/leaderboard?window=final_epoch&sort=income_per_slot
```

## Validator Search

New endpoint:

```text
GET /v1/validators/search?q=<text>&limit=10
```

Search reads only the local database and never calls Solana RPC. It matches
validator name, vote pubkey prefix, identity pubkey prefix, and keybase
username. Opted-out validators are excluded.

## Leader-Slot CU Fields

`GET /v1/validators/{idOrVote}/epochs/{epoch}/leader-slots` now includes
additional additive summary fields derived from stored block facts:

- `costUnits`
- `computeBudgetRequestedUnits`
- `computeBudgetLimitTxCount`
- `computeBudgetPriceTxCount`
- `maxComputeUnitLimit`
- `maxComputeUnitPriceMicroLamports`
- `avgComputeUnitsPerProducedBlock`
- `avgComputeUnitsPerTransaction`
- `avgCostUnitsPerProducedBlock`
- `avgCostUnitsPerTransaction`
- `incomeLamportsPerMillionComputeUnit` / `incomeSolPerMillionComputeUnit`
- `priorityFeeLamportsPerMillionComputeUnit` / `priorityFeeSolPerMillionComputeUnit`
- `tipLamportsPerMillionComputeUnit` / `tipSolPerMillionComputeUnit`

These fields are additive. They do not change existing response fields and do
not increase request-time RPC usage.
